## Vercel Postgres Migration Plan

### 1. Assess Current Data + Traffic
- Core JSON datasets:
  - `data/thinker-source-register.json`: 563 thinker rows, each with `collection`, `thinker`, `slug`, and 1–4 nested `sources` (label, type, url, works_root, notes[]). Generated by `scripts/python/util/build_source_register.py` from crawl outputs; refreshed ad-hoc after data harvests.
  - `data/zero-works-source-matches.json`: 335 thinkers, tracks attempted matches, `status` (`matched`/`unmatched`), and `matches` child array (text, href, url, category). Updated by manual review scripts in `scripts/python/scrapers/`.
  - `data/zero-works-thinkers.json`: 335 entries mirroring the unmatched thinkers list (collection, thinker, slug) for crawl backlog tracking.
  - `public/data-v2/index.json`: 31 category records (`id`, `name`, `path`, `count`) used to enumerate dataset folders.
  - `public/data-v2/<collection>/metadata.json`: per-category thinker metadata with compact keys (`n`, `c`, `d`, `b`, `i`, `t`, `w`, optional `subjects[]`) and derived aggregates (`w` = total works, subject counts).
  - `public/data-v2/<collection>/<Thinker>/<Subject>.json`: lists of individual works (`title`, `url`) grouped by subject; thousands of rows across folders.
- Application consumers:
  - `app/page.tsx` and `components/thinker-search` load all thinker metadata via `loadAllThinkersMetadata()` for global search and landing page content.
  - `app/visualizations/page.tsx` plus D3 components (`WorkCountBarChart`, `WorkCountDistributionChart`) pull aggregated work counts to render charts.
  - Server utilities in `lib/data/folder-loader.ts` dynamically fetch metadata + per-thinker works, expecting the folder hierarchy above.
  - Python ETL scripts (`build_source_register.py`, `scripts/python/scrapers/*`) read/write the JSON files to maintain crawl state.
- Existing derived / aggregated fields:
  - `metadata.json` includes `w` (work count), `subjects[].count`, and `major works` (`j`) for each thinker; these power search filters and chart summaries without re-counting files.
  - Visualizations compute top-50 rankings and histogram buckets from `w`; search expects denormalized thinker cards (name, category, description, URLs, counts).
- Traffic + query profile (based on current usage):
  - Anticipate low concurrency (<50 simultaneous viewers) but heavy read bursts when charts load (`loadAllThinkersMetadata` fetches ~500 records).
  - Majority of requests are read-only; writes happen offline during harvests. Queries are simple filters + aggregations (top-N, distribution histograms).
  - Future growth: more categories and works will inflate JSON bundle sizes, so plan for indexed queries on `collection`, `slug`, `work_count`, and subject tags.

### 2. Draft Schema
- Core tables (Postgres-friendly naming):
  - `collections` (`id` PK, `name`, `display_name`, `description`).
  - `thinkers` (`id` UUID PK, `collection_id` FK, `slug` unique, `name`, `bio_url`, `image_url`, `thumbnail_url`, `description`, `work_count`, `created_at`, `updated_at`).
  - `subjects` (`id` serial PK, `collection_id` FK, `name` unique within collection).
  - `thinker_subjects` (join table: `thinker_id` FK, `subject_id` FK, `work_count`, composite PK).
  - `works` (`id` UUID PK, `thinker_id` FK, `subject_id` FK nullable, `title`, `url`, `source_host`, `published_on` nullable, `ingested_at`).
  - `sources` (`id` UUID PK, `thinker_id` FK, `label`, `source_type`, `url`, `works_root`, `priority`, `notes` JSONB).
  - `zero_work_reviews` (`id` UUID PK, `thinker_id` FK, `status`, `notes`, `last_checked_at`).
  - `zero_work_matches` (`id` UUID PK, `review_id` FK, `text`, `href`, `url`, `category`).
- Indexing + constraints:
  - Unique constraint on (`collection_id`, `slug`) to mirror existing JSON keying.
  - BTREE indexes on `thinkers.collection_id`, `thinkers.work_count`, and `thinker_subjects.subject_id` for chart & filter performance.
  - GIN index on `sources.notes` if notes become non-empty structured data.
  - Foreign keys cascade deletes where appropriate (e.g., removing a thinker cleans up works/matches).
- Views / materialization:
  - Create a view `thinker_work_summary` aggregating `works` counts by thinker and subject to replace current derived JSON totals.
  - Optionally materialize `top_thinkers` (thinker name + work_count) for faster chart queries; refresh nightly if data changes infrequently.
- Documentation:
  - Capture the schema in `docs/schema.md`, including ER diagram and migration order (collections → thinkers → subjects → works → sources → review tables).

```dbml
Table collections {
  id text [pk]
  name text
  display_name text
  description text
  created_at timestamptz
  updated_at timestamptz
}

Table thinkers {
  id uuid [pk]
  collection_id text [ref: > collections.id]
  slug text [unique, not null]
  name text
  bio_url text
  image_url text
  thumbnail_url text
  description text
  work_count int
  created_at timestamptz
  updated_at timestamptz
}

Table subjects {
  id serial [pk]
  collection_id text [ref: > collections.id]
  name text
  created_at timestamptz
  updated_at timestamptz

  Indexes {
    (collection_id, name) [unique]
  }
}

Table thinker_subjects {
  thinker_id uuid [ref: > thinkers.id]
  subject_id int [ref: > subjects.id]
  work_count int
  created_at timestamptz
  updated_at timestamptz

  Indexes {
    (thinker_id, subject_id) [pk]
  }
}

Table works {
  id uuid [pk]
  thinker_id uuid [ref: > thinkers.id]
  subject_id int [ref: > subjects.id, note: 'nullable']
  title text
  url text
  source_host text
  published_on date
  ingested_at timestamptz
  created_at timestamptz
  updated_at timestamptz

  Indexes {
    thinker_id
    subject_id
  }
}

Table sources {
  id uuid [pk]
  thinker_id uuid [ref: > thinkers.id]
  label text
  source_type text
  url text
  works_root text
  priority int
  notes jsonb
  created_at timestamptz
  updated_at timestamptz

  Indexes {
    thinker_id
  }
}

Table zero_work_reviews {
  id uuid [pk]
  thinker_id uuid [ref: > thinkers.id]
  status text
  notes text
  last_checked_at timestamptz
  created_at timestamptz
  updated_at timestamptz
}

Table zero_work_matches {
  id uuid [pk]
  review_id uuid [ref: > zero_work_reviews.id]
  text text
  href text
  url text
  category text
  created_at timestamptz
  updated_at timestamptz
}
```

### 3. Choose Tooling
- Adopt **Drizzle ORM** with the Neon/Vercel Postgres serverless driver for type-safe queries and lightweight bundles.
- Use `drizzle-kit` for schema definition + SQL migration generation; store migrations under `drizzle/migrations/`.
- Add `drizzle.config.ts` (connection settings + schema file paths) and a `lib/db.ts` helper that instantiates the Drizzle client per request using `@neondatabase/serverless`.
- For Python ETL, prefer direct inserts via `psycopg` or CLI-run Drizzle migrations; alternatively expose a small API endpoint if network access is easier than DB connections.

### 4. Provision Vercel Postgres
- In Vercel dashboard, add a Postgres database (Neon-backed). Note connection string, shadow DB URL (for migrations), and serverless driver URL.
- Configure environment variables in Vercel project (`POSTGRES_URL`, `POSTGRES_PRISMA_URL`, etc.) and add `.env.local.example`.
- For local dev, install Neon CLI or use the provided connection string.

### 5. Build Migration + Seed Scripts
- Create initial migration representing the drafted schema.
- Write a Node/Python script that:
  - Reads existing JSON files.
  - Normalizes data into table rows.
  - Uses bulk inserts (COPY or batched INSERTs) to seed the database.
- Ensure script can run idempotently for repeatable environments.

### 6. Implement Data Access Layer
- Create `lib/db.ts` (or similar) to encapsulate DB client initialization using the Vercel serverless driver.
- Add repository/query helpers for each data set (counts, distributions).
- Update API routes or server components to fetch via the new layer.
- Gate the new path behind a feature flag/env to allow gradual rollout.

### 7. Validate & Optimize Queries
- Reproduce existing visualizations, comparing results vs JSON baseline.
- Profile slow queries; add indexes or materialized views as needed.
- Consider precomputing heavy aggregations nightly if needed (stored procedures or cron job on GitHub Actions).

### 8. Update Ops & Tooling
- Add DB migration step to CI (lint → test → `pnpm migrate:check`).
- Document local dev workflow (migrate, seed, reset commands).
- Set up backup strategy (Neon point-in-time restore or scheduled dumps).
- Add monitoring/logging for query errors in Next.js.

### 9. Cutover
- Deploy new API routes/pages using Vercel Postgres.
- Run seed script against production DB.
- Run smoke tests in preview deployment to confirm parity.
- Remove JSON data loading paths once confidence is high; keep archive for rollback.

### 10. Post-Migration Tasks
- Update README/docs with DB requirements and setup steps.
- Schedule periodic vacuum/analyze (Neon handles automatically, but note in docs).
- Plan iterative improvements (e.g., add caching layer, incremental ETL).
